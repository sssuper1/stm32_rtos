# 🚀 软件开发任务书：多级菜单工控屏 (RTOS版)

## 一、 系统整体架构设计 (System Architecture)

系统将基于 STM32 和 RTOS（如 FreeRTOS）构建，划分为三个核心任务（Task）和两个核心组件（Component）。

### 1. 核心任务分配 (RTOS Tasks)

- **交互与显示任务 (HMI_Task)**：
  - **优先级**：中等 (Normal)。
  - **职责**：负责按键扫描（或接收按键中断信号）、驱动 LCD 屏幕刷新、运行菜单状态机。
- **串口通信任务 (UART_Comm_Task)**：
  - **优先级**：高 (High)。
  - **职责**：负责与主控板的 UART 通信。使用 DMA 接收数据，避免 CPU 阻塞；解析协议帧并提取有效数据。
- **网关与业务逻辑任务 (Gateway_Logic_Task)**：
  - **优先级**：低 (Low) 或中等。
  - **职责**：后台处理复杂业务逻辑。如果设备涉及网络网关功能，此任务负责将收到的串口数据进行协议转换（如转换为以太网或网页展示所需格式），并处理定时上报逻辑。

### 2. 任务间通信机制 (IPC)

- **消息队列 (Message Queue)**：HMI 任务修改参数后，将“参数地址+新数值”打包发入队列，UART 任务读取队列并发送。
- **互斥锁 (Mutex)**：保护全局“数据字典”。无论是 HMI 任务想要读取参数显示到屏幕，还是 UART 任务收到新数据想要更新参数，都必须先获取 Mutex，确保数据一致性。

## 二、 阶段开发拆解与实现策略

### 阶段 1：底层驱动与 RTOS 移植 (BSP & OS)

- **诉求**：搭建稳固的软硬件基础环境。
- **实现步骤**：
  1. 配置 STM32 的系统时钟。
  2. 完成 LCD 显示屏驱动（实现画点、清屏、字符串和数字显示函数）。
  3. 完成矩阵键盘或独立按键驱动，加入软件消抖逻辑。
  4. 配置 UART 及 DMA（Idle 中断不定长接收）。
  5. 移植 FreeRTOS，创建并点亮一个测试心跳 LED 任务，确认 OS 调度正常。

### 阶段 2：构建全局“数据字典” (Data Dictionary)

- **诉求**：将 PDF 中的凌乱参数统一化、结构化管理。

- **实现步骤**：

  1. 定义统一的数据结构，涵盖 PDF 中提到的各种类型，例如无符号单字节 `uint1` (0-255) 、有符号单字节 `sint1` (-127~127) 、以及需要处理小数点的 `1000uint4` 等 。

     

     

  2. 创建一个全局的结构体数组 `ParamDef_t Global_Dict[]`，录入所有参数的：地址、当前值、最小值、最大值、小数点位数、读写权限（只读如状态展示，可读写如配置下发）。

### 阶段 3：5级菜单状态机的核心实现 (Menu State Machine)

- 

  **诉求**：彻底抛弃 `if-else` 嵌套，实现层次分明、易于扩展的菜单树。根据需求，一级菜单主要包括“操作”、“功能”、“设置”等 。

  

  

- **每一级的具体实现逻辑**：

  - **数据结构**：采用“节点链表树”结构。每个菜单节点包含：`Menu_ID`, `Parent_ID`, `Child_ID`, `Sibling_ID`, `Render_Callback`, `Key_Callback`。

  - **第 1 级至第 4 级（导航级）**：这些级别主要是目录跳转。

    - **显示实现**：`HMI_Task` 根据当前指针 `Current_Node` 渲染当前同级菜单列表，并高亮光标所在项。
    - **按键实现**：按下“上下”键，指针跳向 `Sibling_ID`；按下“确认”键，指针跳向 `Child_ID`（进入下一级）；按下“返回”键，指针跳向 `Parent_ID`。

  - 

    **第 5 级（叶子节点 / 参数编辑级）**：具体到修改某个参数（如“操作 -> 模式设置 -> 当前工作模式” ）。

    

    

    - **显示实现**：界面切换为“编辑模式”，光标闪烁，提示用户正在修改具体数值。
    - **按键实现**：此时“上下”键的作用变为**增减数值**。每次增减，都会调用数据字典中的 `min_val` 和 `max_val` 进行越界拦截。
    - **确认提交**：按下“确认”键，结束编辑状态，将新数值写入数据字典，并将该操作打包推入 **UART 消息队列**。

### 阶段 4：通信协议栈的闭环 (Protocol Stack)

- **诉求**：工控屏（UI）与主控板（执行者）的数据同步。
- **实现步骤**：
  1. **主动下发**：`UART_Comm_Task` 阻塞等待消息队列。一旦 `HMI_Task` 传来用户修改的参数（如修改了“IP地址”），立即组装串口帧（包头+写指令+地址+数据+校验）并发送。
  2. **被动接收（状态刷新）**：对于需要实时监控的数据（如信号强度、温度），由 `Gateway_Logic_Task` 定时（如每隔1秒）向主控板发送“读指令”。
  3. **解析与更新**：`UART_Comm_Task` 收到主控板的回复后，校验无误，申请 Mutex，将最新数据覆写到“数据字典”中。`HMI_Task` 在下一次屏幕刷新时，自然会读取到最新数据并显示。

------

## 三、 预期交付标准

1. **无阻塞交互**：按键响应时间 < 50ms，屏幕刷新不影响串口数据的持续高频收发。
2. **安全性**：无论用户如何在第5级菜单输入错误格式，均在 UI 层被“数据字典”的极值限制拦截，绝不向主控板发送非法数据。
3. **模块化**：后续若需新增第6级菜单或新增网络通信协议，只需在数组中增加节点或新建 Task，核心状态机代码实现“零修改”。
