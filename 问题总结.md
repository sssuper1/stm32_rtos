# HMI_RTOS_Panel 问题总结（2026-04-01）

## 1. 问题一：WR 参数修改后不上发（长期显示 TX0）

### 1.1 现象
- 操作流程：`*` 进入编辑 -> 修改参数 -> `OK` 确认。
- 预期：屏端组 0x01 写参数帧，通过 UART 上发主控。
- 实际：早期一直显示 `TX0`，串口工具看不到写参数帧。

### 1.2 分阶段定位过程
- 增加链路状态参数：`PARAM_ID_UART_ACK_STATE` + `PARAM_ID_UART_ACK_CMD`。
- 在主界面第 7 行显示 `T/C` 状态（例如 `T1 CB4`）。
- 阶段码定义（关键）：
	- `A1`：进入 `APP_ParamUpdate_RequestValue`。
	- `A2`：请求已入队。
	- `B1`：进入 `UART_SendParamWrite`。
	- `B4`：`APP_UartProto_SendRaw` 成功。
	- `B2/B3`：映射失败 / 发送失败。

### 1.3 根因
- 早期链路偏“队列异步优先”，请求停在 `A2` 时，用户可见层面像“没发包”。
- 部分提交逻辑曾依赖 `TrySetValue` 成功后才触发上发，存在“可编辑但提交不上发”的窗口。
- 发送阶段状态未即时刷新到 UI，导致排障时看起来长期停在旧状态。

### 1.4 修复项
1. 菜单提交路径统一改为 `OK` 后直接调用 `APP_ParamUpdate_RequestValue(id, applyVal)`。
2. `APP_ParamUpdate_RequestValue` 改为“同步直发优先，队列重试兜底”。
3. `UART_SendParamWrite` 增加阶段码，失败/成功都会回写状态。
4. 发送后立刻 `APP_Menu_RefreshCurrent()`，保证主界面状态实时变化。
5. `BSP_Uart_Send` 增加 3 次重试，降低 HAL BUSY 造成的偶发丢发。

### 1.5 关键代码锚点
- 菜单 OK 提交上发：`Core/Src/app_menu.c`（2818, 2905, 3123, 3257, 3360）
- 请求入口与阶段码：`Core/Src/freertos.c`（117, 124, 125, 129, 140）
- 发送函数与阶段码：`Core/Src/freertos.c`（410, 419, 433, 471, 479）
- 发送后刷新界面：`Core/Src/freertos.c`（435, 473, 481）
- TX 重试：`Core/Src/bsp_stm32.c`（58, 60）
- 主界面显示 T/C：`Core/Src/app_menu.c`（1644, 1645, 1688）

### 1.6 验收结果
- 最终现场状态：`T1 CB4`。
- 串口工具可见写参数帧输出，问题闭环。

## 2. 问题二：串口下发后 main 界面无更新

### 2.1 现象
- 主控下发 0x04/0x05/0x08 等状态后，屏幕主界面字段不变化或更新不完整。

### 2.2 根因
- 部分命令解析字段覆盖不完整，导致主界面读取到旧值。
- 数据字典更新后未统一触发菜单刷新时，当前页无法即时重绘。

### 2.3 修复项
1. 补齐 0x05 完整 DeviceConfig（61B）解析并写回字典（设备名 token、多网口、串口格式、自动同步、时间等）。
2. 对 0x08 主界面相关字段按协议顺序完整落库。
3. 对 0x02/0x04/0x05/0x06/0x07/0x08/0x09/0x0A 处理后统一调用 `APP_Menu_RefreshCurrent()`。
4. 主界面渲染统一从字典拉取当前值，不缓存旧值作为显示源。

### 2.4 关键代码锚点
- 0x05 解析入口：`Core/Src/app_uart_handlers.c`（417）
- 0x08 解析入口：`Core/Src/app_uart_handlers.c`（320 附近）
- 多命令处理后刷新：`Core/Src/app_uart_handlers.c`（307, 409, 509, 563, 632, 727, 811, 893）
- 主界面按字典渲染：`Core/Src/app_menu.c`（1623-1692 区域）

### 2.5 验收结果
- 下发后主界面时间、波形、带宽、跳频、频点、空域滤波、同步模式、功率/衰减可联动刷新。

## 3. 排障经验沉淀

1. 对“看起来没发包”的问题，优先加链路阶段码（请求入口/入队/发送入口/发送结果），避免只看最终现象。
2. 菜单提交路径优先使用显式值上发（`RequestValue`），不要依赖“先写字典成功再上发”的单点前置条件。
3. 串口接收解析后必须统一刷新当前页，否则会出现“字典已更新但 UI 不更新”的假象。
4. UART 物理发送建议保留轻量重试，降低 HAL BUSY 引发的瞬时失败。

## 4. 回归测试清单

1. 任一 WR 项：`*` -> 修改 -> `OK` 后检查 `T/C` 最终到 `T1 CB4`。
2. 抓包确认 0x01 帧：`D5 5D 01 FF ... CRC ... 5D D5`。
3. 下发 0x08 后检查 main 页面全部关键字段即时更新。
4. 下发 0x05（61B）后检查设置页与 main 页相关字段联动更新。
5. 连续快速修改多个 WR 项，确认无卡死、无长时间停留 `A2/B1`。
