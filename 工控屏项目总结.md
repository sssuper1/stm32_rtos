# HMI_RTOS_Panel Core 项目总结（细节版，含工作量拆分）

> 目标：在“框架版”基础上补齐可落地细节，明确区分哪些是业务自研工作量，哪些是 CubeMX 生成骨架。
> 适用范围：`projects/HMI_RTOS_Panel/Core` 下 `Src` 与 `Inc`。

---

## 一、系统启动与运行主链（细节）

### 1. 启动时序

`main()` 的实际运行时序可归纳为：

1. `HAL_Init()` 初始化 HAL、Systick 基础。
2. `SystemClock_Config()` 完成系统时钟配置。
3. `MX_GPIO_Init()`、`MX_DMA_Init()`、`MX_USART1_UART_Init()`、`MX_I2C2_Init()` 初始化外设。
4. `osKernelInitialize()` 初始化 CMSIS-RTOS2 内核。
5. `MX_FREERTOS_Init()` 创建互斥锁、队列和线程。
6. `osKernelStart()` 启动调度，进入任务态。

当前工程中 `MX_SPI1_Init()` 在 `main.c` 内被注释，说明主链路以“按键 + OLED(I2C) + USART1(DMA)”为核心。

### 2. RTOS 对象拓扑

`freertos.c` 中运行时关键对象：

1. `gParamMutexHandle`：参数字典互斥锁。
2. `gParamUpdateQueueHandle`：参数上报队列（菜单线程到串口线程）。
3. `HMI_Task`（`StartDefaultTask`）：UI 输入和显示。
4. `UART_Task`（`StartTask02`）：协议接收与队列补发。
5. `Logic_Task`（`StartTask03`）：轻逻辑与接收兜底。

这套拓扑把“交互”和“收发”解耦：HMI 只负责触发业务意图，UART 负责可靠下发。

### 3. USART1 DMA + IDLE 接收底座

收包底座由 `usart.c` 和 `stm32f1xx_it.c` 共同构成：

1. `HAL_UART_Receive_DMA()` 开启 USART1 DMA 循环接收。
2. `__HAL_UART_ENABLE_IT(UART_IT_IDLE)` 开启 IDLE 中断用于分帧触发。
3. `APP_USART1_RxCheckIdle()` 读取 DMA 游标增量并推送到 `APP_UartRx_PushFromISR()`。
4. `USART1_IRQHandler` 与 `DMA1_Channel5_IRQHandler` 都会触发 `APP_USART1_RxCheckIdle()`，降低漏字节风险。

## 二、业务核心调用链

### 1. WR 参数上报链（OK 提交）

完整链路：

`按键事件 -> APP_Menu_HandleKey -> MenuKeyHandler_* -> APP_ParamUpdate_RequestValue -> UART_SendParamWrite -> APP_UartProto_SendRaw(cmd=0x01,type=0xFF) -> BSP_Uart_Send`

关键机制：

1. 提交值采用 `RequestValue(id, applyVal)`，避免依赖“先从字典读回再发送”的脆弱路径。
2. `APP_ParamUpdate_RequestValue` 采用“直发优先 + 队列兜底”：
	- 先同步调用 `UART_SendParamWrite`。
	- 失败时再入队，交给 `UART_Task` 异步补发。
3. `UART_SendParamWrite` 负责把 `ParamId` 映射为主控地址和字节长度。
4. `BSP_Uart_Send` 增加 3 次重试，提升 HAL BUSY 场景稳定性。

### 2. 下行状态刷新链（主控回包到界面）

完整链路：

`USART1 DMA/IDLE -> APP_UartRx_PushFromISR -> APP_UartRx_ProcessPending -> APP_UartRx_OnByte(FSM) -> APP_UartRx_OnFrame -> handle_cmd_0x** -> APP_ParamDict_SetValueUnsafe -> APP_Menu_RefreshCurrent`

关键机制：

1. `app_uart_rx.c` 仅处理传输层状态机（帧头帧尾、CRC、粘包半包）。
2. `app_uart_handlers.c` 负责命令语义（`0x02/0x04/0x05/0x06/0x07/0x08/0x09/0x0A`）。
3. 命令处理后统一刷新菜单，解决“已更新字典但界面未刷新”的观感断层。

### 3. 诊断可视化链（T/C 状态码）

主界面展示 `UART_ACK_STATE`/`UART_ACK_CMD`，用于快速现场定位：

1. 请求阶段：`A1/A2/A3/A4`。
2. 发送阶段：`B1/B2/B3/B4`。
3. 发送或失败后触发 `APP_Menu_RefreshCurrent()`，确保诊断码可见。

## 三、线程职责边界

### 1. HMI_Task

1. 执行 `BSP_Lcd_Init()`、`BSP_Keys_Init()`、`APP_Menu_Init()`。
2. 读取按键并驱动菜单状态机。
3. 执行空闲超时回主页（60s）。
4. 兜底调用 `APP_UartRx_ProcessPending()`。

### 2. UART_Task

1. 周期消费串口 pending 字节。
2. 从 `gParamUpdateQueueHandle` 取 `ParamUpdateMsg_t` 并发送。
3. 承担同步直发失败后的补发职责。

### 3. Logic_Task

1. 轻逻辑循环与可选诊断发送。
2. 再次调用 `APP_UartRx_ProcessPending()`，形成第三条收包处理路径。

## 五、工作量拆分（自己的工作量 vs CubeMX 生成）

以下拆分依据两点：

1. 文件来源是否为 CubeMX 典型模板。
2. 是否落在 `USER CODE BEGIN/END` 区域内承载业务逻辑。

### A. 纯业务自研（主要工作量）

1. `Core/Src/app_menu.c`
2. `Core/Src/app_param_dict.c`
3. `Core/Src/app_param_uart_map.c`
4. `Core/Src/app_uart_handlers.c`
5. `Core/Src/app_uart_proto.c`
6. `Core/Src/app_uart_rx.c`
7. `Core/Src/bsp_keys.c`
8. `Core/Src/bsp_oled.c`
9. `Core/Src/bsp_stm32.c`
10. 对应 `Core/Inc` 下 `app_*`、`bsp_*` 头文件。

这部分承担了菜单交互、协议栈、参数模型、硬件抽象、诊断机制等核心能力，属于项目价值主体。

### B. CubeMX 生成骨架 + 在 USER CODE 中二次开发（次核心工作量）

1. `Core/Src/freertos.c`
	- 业务增量：任务内业务循环、参数更新请求链、直发优先/队列兜底策略、阶段码。
2. `Core/Src/usart.c`
	- 业务增量：DMA+IDLE 接收联动、`APP_USART1_RxCheckIdle` 相关逻辑。
3. `Core/Src/stm32f1xx_it.c`
	- 业务增量：在中断路径注入 `APP_USART1_RxCheckIdle()`。
4. `Core/Src/main.c`
	- 业务增量：启动顺序中的工程化拼接与外设启停取舍（如 SPI 注释）。

这部分体现了“用生成框架承载业务”的工程能力，技术工作量不可忽略。

### C. 以 CubeMX/HAL 模板为主（基础设施工作量）

1. `Core/Src/gpio.c`
2. `Core/Src/dma.c`
3. `Core/Src/i2c.c`
4. `Core/Src/spi.c`
5. `Core/Src/stm32f1xx_hal_msp.c`
6. `Core/Src/stm32f1xx_hal_timebase_tim.c`
7. `Core/Src/system_stm32f1xx.c`
8. `Core/Src/syscalls.c`
9. `Core/Src/sysmem.c`
10. 大部分 HAL/启动配置头文件。

这部分主要负责芯片启动和外设底层模板，不是业务复杂度来源，但决定系统可运行性。

### D. 结论（工作量归因）

1. 业务复杂度主要集中在 `app_*` + `bsp_*` + `freertos.c` 的 USER CODE 业务区。
2. CubeMX 贡献了“可编译、可启动、可管理”的平台骨架。
3. 实际项目产出价值主要来自自研业务层与在骨架中的二次开发整合。

## 六、关键问题闭环（本阶段）

### 1. WR 参数 OK 后不上报

根因链：

1. 提交路径依赖字典中间态，导致“可编辑但不一定触发有效发送”。
2. 仅队列发送时，现场诊断不直观。
3. 个别参数权限配置导致提交被拦截。

修复链：

1. 统一改为 `APP_ParamUpdate_RequestValue(id, applyVal)`。
2. 请求策略改为“直发优先，队列兜底”。
3. UART 发送增加 3 次重试。
4. 经纬高参数权限改 RW。
5. 增加 T/C 诊断码并强制刷新页面。

结果：现场状态达到 `T1 CB4`，串口可见发送包。

### 2. 主控下发后 main 页面无更新

根因链：

1. 下行回包写字典后，界面刷新触发时机不一致。

修复链：

1. 在关键命令处理后统一 `APP_Menu_RefreshCurrent()`。
2. 在发送状态变化后也触发刷新，保证诊断信息同步显示。

结果：形成“回包 -> 写字典 -> 刷新显示”稳定闭环。

## 七、回归检查清单（建议长期保留）

1. 改任意 WR 参数并按 OK，需看到 `T1` 和 `CB4`（或等价成功状态）。
2. 串口抓包应出现 `0x01/0xFF` 写参数帧，CRC 正确。
3. 下发 `0x02~0x0A` 后，菜单/主页显示应在可接受周期内更新。
4. 将 UART 置于 BUSY 压测场景，验证重试策略仍可成功发送。
5. 回归时避免修改 CubeMX 文件的 USER CODE 外区域，防止重新生成覆盖。

---

## 八、维护建议

1. 新增参数时按“字典 -> 映射 -> 菜单 -> 协议”四步走，避免链路断点。
2. 保留并复用 T/C 诊断码体系，优先用于现场定位。
3. 所有可编辑参数必须在字典中明确 RW 与边界。
4. 若后续重新 CubeMX 生成代码，先备份 `freertos.c/usart.c/stm32f1xx_it.c/main.c` 的 USER CODE 业务块。

本文件作为 Core 层的“结构+责任+工作量”基线，后续可继续按命令号、参数组、页面组做更细粒度附录。

## 九、文件级实现展开（业务核心层 + BSP 适配层）

本节针对“模块分层与文件职责”中的每个业务文件，补充“实现框架 + 核心函数 + 调用关系”。

### 1. app_menu.c / app_menu.h

实现框架：

1. 以 `MenuNode_t` 树结构组织 3 大主菜单与多级子菜单。
2. 通过 `APP_Menu_HandleKey` 驱动统一按键状态机。
3. 对“可编辑叶子节点”通过 `key_cb` 拦截进入编辑态。
4. 渲染由 `render_cb` 解耦，支持根页综合状态显示与子页列表显示。

关键函数：

1. `APP_Menu_Init`：初始化当前节点、编辑状态，首次渲染。
2. `APP_Menu_HandleKey`：先执行节点 `key_cb`，未拦截时走默认导航。
3. `APP_Menu_DefaultRender`：根页渲染（时间、入网、频点、功率、T/C 诊断）与通用页渲染。
4. `Render_OperationTree`：7 行分页展示兄弟节点，并对选中项附带参数值。
5. `Render_MemberNodeList`：显示 0x09 邻居缓存列表并高亮当前选择。
6. `MenuKeyHandler_OperationItem`：通用 WR 参数编辑（`*` 进编辑，`OK` 提交）。
7. `MenuKeyHandler_TrafficOp`：流量操作三态（START/STOP/CLEAR）编辑与提交。
8. `MenuKeyHandler_UartBaud`：波特率索引编辑并提交 `PARAM_ID_UART_BAUD_IDX`。
9. `MenuKeyHandler_UartFormat`：数据位/停止位/校验/流控索引编辑与提交。
10. `MenuKeyHandler_TimeManual`：时分秒编辑并提交。
11. `MenuKeyHandler_MemberNodeList` / `MenuKeyHandler_DetailRequest`：按选中节点发起 `0x0A` 详情请求。

调用关系（核心）：

1. `APP_Menu_HandleKey -> MenuKeyHandler_* -> APP_ParamDict_TrySetValue -> APP_ParamUpdate_RequestValue`（上报链）。
2. `APP_Menu_RefreshCurrent <- app_uart_handlers.c`（下行更新后刷新链）。

### 2. app_param_dict.c / app_param_dict.h

实现框架：

1. 以 `s_paramDict[]` 作为全局参数单一事实源。
2. 参数项包含类型、权限、边界、scale、当前值。
3. 对外暴露“按 ID 查找 / 读取 / 安全写 / 非安全写”接口。

关键函数：

1. `APP_ParamDict_GetTable`：导出参数表与总数（调试/遍历）。
2. `APP_ParamDict_FindById`：按 `ParamId_t` 线性查找参数项。
3. `APP_ParamDict_GetValue`：互斥保护下读取参数值。
4. `APP_ParamDict_TrySetValue`：带权限和边界检查的写入。
5. `APP_ParamDict_SetValueUnsafe`：跳过 RW 权限，仅保留边界检查，供回包更新使用。

调用关系（核心）：

1. 上行编辑提交调用 `TrySetValue`。
2. 下行协议处理调用 `SetValueUnsafe`。

### 3. app_param_uart_map.c / app_param_uart_map.h

实现框架：

1. `s_writeMap[]` 集中定义 `ParamId -> 主控地址 + valueLen`。
2. 写参数发送前统一走该映射，避免业务层散落硬编码地址。

关键函数：

1. `map_find`：查找参数映射并校验地址与长度合法性。
2. `APP_ParamUartMap_BuildWrite`：输出地址、值字节数组和长度（1/2/4 字节编码）。

调用关系（核心）：

1. `freertos.c:UART_SendParamWrite -> APP_ParamUartMap_BuildWrite -> APP_UartProto_SendRaw`。

### 4. app_uart_proto.c / app_uart_proto.h

实现框架：

1. 只负责协议帧封装与 CRC，不处理业务语义。
2. 固定头尾：`D5 5D` / `5D D5`。
3. CRC16(A001) 覆盖 `cmd + type + payload`。

关键函数：

1. `APP_UartProto_CalcCrc`：CRC16 计算并做高低字节交换。
2. `APP_UartProto_BuildFrame`：拼接完整帧并写入输出缓冲。
3. `APP_UartProto_SendRaw`：封装后调用 `BSP_Uart_Send` 发出。

### 5. app_uart_rx.c / app_uart_rx.h

实现框架：

1. ISR 与任务分离：ISR 只进环形缓冲，任务态做协议解析。
2. 接收状态机按帧头、帧尾、CRC完成帧判定。
3. 统计多个诊断计数（head/tail/crc fail/valid），支持现场定位。

关键函数：

1. `APP_UartRx_Init`：清空状态机、ring 和统计计数。
2. `APP_UartRx_PushFromISR`：中断上下文推字节到 ring。
3. `APP_UartRx_ProcessPending`：任务态取 ring 字节并喂给 FSM。
4. `APP_UartRx_OnByte`：核心 FSM，发现完整帧后校验 CRC。
5. `APP_UartRx_OnFrame`（weak）：默认弱符号回调，由 `app_uart_handlers.c` 实现覆盖。
6. `APP_UartRx_Get*`：导出有效帧数、CRC 失败数、帧头帧尾命中等指标。

### 6. app_uart_handlers.c

实现框架：

1. 承接 `APP_UartRx_OnFrame`，按 `cmd` 分发到 `handle_cmd_0x**`。
2. 每个命令负责 payload 解码、字典写回、页面刷新。
3. 对 0x09 维护邻居缓存，支持菜单按索引切换与详情请求。

关键函数：

1. `APP_UartRx_OnFrame`：分发 `0x02/0x04/0x05/0x06/0x07/0x08/0x09/0x0A`。
2. `handle_cmd_0x02`：ACK 状态写回 `UART_ACK_STATE/UART_ACK_CMD`。
3. `handle_cmd_0x04`：链路参数（模式/协议/频点/带宽/功率）写回。
4. `handle_cmd_0x05`：设备基础信息写回（兼容新版完整帧与旧版简化帧）。
5. `handle_cmd_0x06`：统计数据写回（兼容 12B 新版与 16B 旧版）。
6. `handle_cmd_0x07`：自检状态写回。
7. `handle_cmd_0x08`：主页显示参数写回。
8. `handle_cmd_0x09`：邻居列表解析入缓存并更新当前显示节点。
9. `handle_cmd_0x0A`：节点详情写回。
10. `APP_UartRx_GetNeighborCountCached` / `APP_UartRx_GetNeighborNodeIdByIndex` / `APP_UartRx_LoadNeighborByIndexToParams`：菜单层读取邻居缓存能力。
11. `parse_ascii_coord_1e6`、`read_be_u32`、`read_le_u16` 等：协议字段解码工具函数。

调用关系（核心）：

1. `app_uart_rx.c` 完整帧 -> `APP_UartRx_OnFrame`。
2. `APP_UartRx_OnFrame` -> `handle_cmd_0x**` -> `APP_ParamDict_SetValueUnsafe` -> `APP_Menu_RefreshCurrent`。

### 7. app_param_update.h（接口职责）

实现框架：

1. 该头文件定义 HMI 侧参数上报接口。
2. 实现在 `freertos.c`（请求队列与直发策略）。

关键接口：

1. `APP_ParamUpdate_Request`：按字典当前值请求上报。
2. `APP_ParamUpdate_RequestValue`：按显式值请求上报（当前主链路）。

### 8. bsp_stm32.c + bsp.h

实现框架：

1. 统一抽象 LED/LCD/按键/UART，屏蔽 HAL 细节。
2. 将 OLED 与键盘驱动对上层暴露为通用 BSP 接口。

关键函数：

1. `BSP_Led_Set`：板级 LED 控制。
2. `BSP_Lcd_Init` / `BSP_Lcd_PrintLine`：转发到 OLED 驱动。
3. `BSP_Keys_Init` / `BSP_Keys_GetEvent`：转发到键盘驱动。
4. `BSP_Uart_Init`：动态改波特率并重初始化 UART。
5. `BSP_Uart_Send`：UART 发送并做 3 次 BUSY 重试。

### 9. bsp_keys.c / bsp_keys.h

实现框架：

1. 4x4 矩阵键盘扫描（列驱动 + 行读取）。
2. 去抖策略：`rawMask` 稳定 `KEY_DEBOUNCE_MS` 后再认定事件。
3. 使用 `s_keyMap` 完成硬件行列到逻辑 `KeyCode` 映射。

关键函数：

1. `BSP_KeysDrv_Init`：GPIO 配置与状态初始化。
2. `Keys_ScanRawMask`：一次完整矩阵扫描。
3. `BSP_KeysDrv_GetEvent`：比较稳定状态并输出按下/释放事件。

### 10. bsp_oled.c / bsp_oled.h

实现框架：

1. 双缓冲 + 脏页刷新：`front/back + dirtyMask`。
2. 字模渲染采用 6x8 固定字库，按行输出。
3. I2C 发送分片（chunk）以适配总线传输稳定性。

关键函数：

1. `BSP_Oled_Init`：执行 SSD1306 初始化序列并清屏。
2. `BSP_Oled_PrintLine`：文本转字模写入 back buffer。
3. `oled_mark_dirty_if_changed` / `oled_flush_dirty_pages`：仅刷新变化页面。
4. `oled_write_cmd` / `oled_write_data`：底层 I2C 命令与数据发送。

## 十、按实现框架阅读代码的建议顺序

1. 先看 `app_menu.c`（用户输入与上报触发点）。
2. 再看 `app_param_dict.c` + `app_param_uart_map.c`（参数语义与地址映射）。
3. 然后看 `app_uart_proto.c` + `app_uart_rx.c` + `app_uart_handlers.c`（协议收发闭环）。
4. 最后看 `bsp_stm32.c` + `bsp_keys.c` + `bsp_oled.c`（硬件适配细节）。

按上述顺序，可把“按键 -> 参数 -> 发包 -> 回包 -> 刷新显示”完整链路在一次阅读中串起来。
